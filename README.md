# Gitea on OpenShift

Self-hosted Git service deployed on OpenShift via Argo CD GitOps.

## Environments

| Environment | Namespace | URL | Sync |
|-------------|-----------|-----|------|
| Production | `gitea` | https://gitea.home.mburnsfire.net | Manual |
| Staging | `gitea-staging` | https://gitea-staging.home.mburnsfire.net | Automatic |

## Architecture

- **Gitea**: 3 replicas (production), 1 replica (staging)
- **PostgreSQL HA**: 3 PostgreSQL + 2 Pgpool (production); single instance (staging)
- **Redis/Valkey**: standalone for sessions, cache, queue
- **Storage**: NFS-backed RWX for Gitea data, RWO for databases
- **SSH**: NodePort 32222 (production), 32223 (staging)
- **Access**: via Nginx Proxy Manager (NPM), not OpenShift Routes

## Helm Chart

- **Chart**: `https://dl.gitea.com/charts/` — `gitea` chart
- **Version**: `12.5.0` (designed for gitea 1.25.x)
- **Running**: gitea 1.26.1 (see Known Issues below)

## RBAC Requirements

Both namespaces need anyuid SCC and Argo CD admin access before Argo CD can deploy:

```bash
# Grant anyuid SCC (required for NFS write permissions)
oc adm policy add-scc-to-user anyuid -z default -n <namespace>

# Grant Argo CD admin access
oc create rolebinding argocd-admin \
  --clusterrole=admin \
  --serviceaccount=openshift-gitops:openshift-gitops-argocd-application-controller \
  -n <namespace>
```

## Secrets

Secrets are **git-ignored** and must be applied manually. See `staging/secrets.yaml.example` and `production/values-secrets.yaml.example` for templates.

Sensitive values for production (Authentik OAuth2 `CLIENT_ID`/`CLIENT_SECRET`) are applied via:
```bash
helm upgrade gitea ... -f production/values.yaml -f production/values-secrets.yaml
```

## Known Issues

### chart 12.5.0 incompatible with gitea 1.26.x

The `config_environment.sh` script generated by chart 12.5.0 calls `environment-to-ini` with flags that changed in gitea 1.26:

| Version | Old (broken) | Correct |
|---------|-------------|---------|
| gitea ≤1.25 | `environment-to-ini -o $GITEA_APP_INI` | — |
| gitea 1.26+ | — | `environment-to-ini --config $GITEA_APP_INI --apply-env --out $GITEA_APP_INI` |

**After every new deployment or if pods fail to start**, patch the generated secret:

```bash
# For staging:
CURRENT=$(oc get secret -n gitea-staging gitea-staging -o jsonpath='{.data.config_environment\.sh}' | base64 -d)
PATCHED=$(echo "$CURRENT" | sed 's|environment-to-ini.*\$GITEA_APP_INI|environment-to-ini --config $GITEA_APP_INI --apply-env --out $GITEA_APP_INI|g')
oc patch secret gitea-staging -n gitea-staging --type='json' \
  -p="[{\"op\":\"replace\",\"path\":\"/data/config_environment.sh\",\"value\":\"$(echo "$PATCHED" | base64 -w0)\"}]"

# For production (same pattern, different secret/namespace):
CURRENT=$(oc get secret -n gitea gitea -o jsonpath='{.data.config_environment\.sh}' | base64 -d)
PATCHED=$(echo "$CURRENT" | sed 's|environment-to-ini.*\$GITEA_APP_INI|environment-to-ini --config $GITEA_APP_INI --apply-env --out $GITEA_APP_INI|g')
oc patch secret gitea -n gitea --type='json' \
  -p="[{\"op\":\"replace\",\"path\":\"/data/config_environment.sh\",\"value\":\"$(echo "$PATCHED" | base64 -w0)\"}]"
```

The staging Argo CD app has `ignoreDifferences` on this field so Argo CD won't revert it. Production does not — apply the patch after any Argo CD sync that touches the secret.

This will be resolved when a chart version compatible with gitea 1.26 is released.

## Bootstrapping a New Environment

1. **Create namespace and apply RBAC** (see [RBAC Requirements](#rbac-requirements))

2. **Create secrets** — copy and fill in `staging/secrets.yaml.example`
   ```bash
   oc apply -f staging/secrets.yaml  # git-ignored, apply manually
   ```

3. **Apply Argo CD application**
   ```bash
   oc apply -f argocd/gitea-staging.yaml
   ```

4. **Patch `config_environment.sh`** (see [Known Issues](#known-issues)) — required before pods can start on gitea 1.26+

## Monitoring

```bash
# Argo CD status
oc get application -n openshift-gitops | grep gitea

# Pod status
oc get pods -n gitea
oc get pods -n gitea-staging

# Logs
oc logs -n gitea -l app.kubernetes.io/name=gitea --tail=100
```

## Prod Rollback

```bash
# Remove Argo CD management without touching running resources
oc delete application gitea-prod -n openshift-gitops

# Roll back Helm release
helm rollback gitea 25 -n gitea  # update revision number after each sync
```
